# 计划：修复 NMC 台风时间 8 小时偏差（UTC 误作北京时间）

## 摘要

用户指出：巴威（BAVI, 2026）台风登陆时间官方约在「晚上12点左右」，但应用显示 7/11 17:00，怀疑路径有误。

**排查结论**：
- **台风路径（坐标）正确** —— 巴威登陆点 28.5°N 120.8°E（浙江温州/台州一带），`detectLandfall` 海→陆穿越检测选中的点位置正确。
- **时间显示错误 8 小时** —— NMC 路径点的 `p[1]` 时间字段为 **UTC**（经 `p[2]` 时间戳与 `p[12]` 显示字段三方核实），而 `nmcTime()`（[index.html:2619](file:///d:/kzht/Documents/typhoon-fujian-tracker/index.html#L2619)）直接按字面格式化、未做时区转换，导致所有时间显示为 UTC（比北京时间晚 8h）。
- 巴威登陆点 `p[1]="202607111700"`（UTC 17:00）= **北京时间 7/12 01:00**（凌晨刚过午夜）——与用户「晚上12点左右」吻合。

这是**全局时间显示 bug**，影响所有台风（1949 至今）的所有时间字段：观测时次、登陆时刻、最近经过时刻、逐小时表、预报最近点、生命周期时间线等。非路径错误。

## 根因（三方证据核实 p[1] 为 UTC）

NMC `view_{id}` 每个路径点：`[id, p1时间串, p2时间戳ms, 等级, 经度, 纬度, 气压, 风速, 移向, 移速, 风圈, 预报, p12显示]`

以巴威首点为例：
| 字段 | 值 | 含义 |
|------|-----|------|
| `p[1]` | `"202607020000"` | 字面 00:00 |
| `p[2]` | `1782950400000` ms | `new Date(ts).toISOString()` = `2026-07-02T00:00:00.000Z`（**UTC 00:00**）|
| `p[12]` | `"2026年07月02日08时00分"` | NMC 官网显示 = **北京 08:00** |

- `p[2]` 时间戳 = UTC 00:00，`p[1]` 字面 = 00:00 → **p[1] 即 UTC**。
- `p[12]` = 08:00 = UTC+8 → NMC 官网用北京时间显示，`p[1]` 是其 UTC 内部值。
- 桑美（2006）同样：`p[1]="200608050000"`、`p[2]=1154736000000` = UTC 00:00 → 历史台风 p[1] 亦为 UTC，**修复跨年份通用**。
- `p[12]` 仅近期台风有值（历史台风为 null），不能作为通用数据源；`p[1]`+8h 才是通用解法。

## 现状分析（时间处理链）

- **`nmcTime(p[1])`**（L2619-2623）：把 `"202607020000"` 格式化为 `"2026-07-02 00:00:00"` 字符串，**无时区转换**。因 p[1] 为 UTC，输出即 UTC，比北京晚 8h。所有 `point.time`/`forecast.time` 均由此产生。
- **`parseTyphoonTime(str)`**（L3699-3705）：取 nmcTime 输出串，`Date.UTC(y,mo-1,d,H-8,M)`——假设输入为北京时、减 8h 得 UTC 时刻。但因输入实为 UTC，此函数把时刻又往前推 8h，与 `new Date()`（真实当前时刻）比较时，"今天/明天"判定、未来小时筛选等均偏 8h（影响活动台风的逐小时表/预报标签）。
- `detectLandfall`（L3376）纯坐标检测，不涉时间运算，选点正确；仅返回的 `time` 字符串是 UTC，显示偏 8h。
- 所有时间展示（`fmtPointTime`/`shortTimeLabel`/登陆情况/逐小时表/历史影响卡）均消费 `point.time`，修 `nmcTime` 即全链路修正。

## 修改方案

### 步骤 1：`nmcTime` 改为 UTC→北京(UTC+8) 转换

**文件**：`index.html` L2619-2623

**改动**：用 `Date.UTC` 构造 p[1] 的 UTC 时刻，加 8h 后按 UTC 分量格式化（Date 算术自动处理跨日/跨月/跨年）。

```js
function nmcTime(str) {
  const m = String(str || "").match(/^(\d{4})(\d{2})(\d{2})(\d{2})(\d{2})$/);
  if (!m) return String(str || "");
  // NMC p[1] 为 UTC 时刻（经 p[2] 时间戳与 p[12] 显示字段核实），转北京时(UTC+8)后按分量格式化；
  // Date 算术自动处理跨日/跨月/跨年（如 UTC 18:00 +8h → 次日 02:00）。
  const d = new Date(Date.UTC(+m[1], +m[2] - 1, +m[3], +m[4], +m[5]) + 8 * 3600 * 1000);
  const p = (n) => String(n).padStart(2, "0");
  return `${d.getUTCFullYear()}-${p(d.getUTCMonth() + 1)}-${p(d.getUTCDate())} ${p(d.getUTCHours())}:${p(d.getUTCMinutes())}:00`;
}
```

**为何不动 `parseTyphoonTime`**：其 `H-8` 恰好把 nmcTime 输出的北京时（已 +8h）还原为真实 UTC 时刻，与 `new Date()` 比较正确。保持不变即与修复后的 nmcTime 自洽（已验证往返）：
- p[1]=`"202607020000"`（UTC 00:00，实际北京 08:00）
- 修复后 nmcTime → `"2026-07-02 08:00:00"`（北京）✓
- parseTyphoonTime → `Date.UTC(2026,6,2,8-8,0)` = `2026-07-02 00:00 UTC` = 北京 08:00 ✓（真实时刻）

## 假设与决策

1. **p[1] 为 UTC**：经 `p[2]` 时间戳（`toISOString` = UTC）与 `p[12]`（北京显示）三方核实，跨 2026/2006 两个年份一致，确认通用。
2. **不改 `parseTyphoonTime`**：其 `H-8` 与修复后 nmcTime 的 `+8h` 互逆，往返自洽；改动它反而引入新不一致。
3. **不用 `p[12]` 作数据源**：历史台风 `p[12]` 为 null，不可用；`p[1]`+8h 通用。
4. **不动 build 脚本 / typhoon-index.json**：构建与索引不存储/使用时间，仅前端展示受影响。
5. **路径坐标无误**：用户疑虑"路径是否错误"——坐标正确，仅时间显示偏差；登陆点选取（detectLandfall）正确。
6. **影响范围**：仅 `index.html` 一个函数（nmcTime），全链路时间字段自动修正。
